# 问题记录 3: CI/CD 流水线与 Docker 镜像构建问题

## 1. 基础镜像拉取失败 (Pull Access Denied)

**问题现象**:
在 GitHub Actions 的 `CD - Build & Deploy` 工作流中，构建应用镜像时报错：
```
ERROR: failed to solve: ljk4/cloud-robot-viz-base:latest: pull access denied, repository does not exist or may require authorization: server message: insufficient_scope: authorization failed
```

**原因分析**:
1.  **镜像未发布**: 虽然我们在本地可能有这个镜像，但从未推送到 Docker Hub，GitHub Actions 的云端 Runner 无法拉取。
2.  **用户名硬编码错误**: `Dockerfile` 中直接写死使用了 `ljk4` 作为用户名 (`FROM ljk4/...`)，但实际配置的 GitHub Secret `DOCKER_USER` 是 `liujiankang123`。导致权限验证失败或找不到镜像。

**解决方案**:
1.  **新建基础镜像构建流**: 创建 `.github/workflows/ci-base-image.yml`，专门用于构建并推送基础镜像到 Docker Hub。
2.  **参数化 Dockerfile**: 修改 `docker/Dockerfile`，不再硬编码用户名，而是通过 `ARG` 接收参数。
    ```dockerfile
    ARG DOCKER_USER=liujiankang123
    FROM ${DOCKER_USER}/cloud-robot-viz-base:latest AS builder
    ```
3.  **传递构建参数**: 在 `cd-deploy.yml` 中配置 `build-args`，将 Secrets 中的用户名传入 Docker 构建环境。

---

## 2. 新增 Workflow 在 Actions 界面不可见

**问题现象**:
创建了 `ci-base-image.yml` 并推行到 `dev` 分支，但在 GitHub 仓库的 "Actions" 标签页中找不到该工作流，无法手动触发运行。

**原因分析**:
GitHub Actions 的安全机制规定：带有 `workflow_dispatch` (手动触发) 事件的工作流文件，**必须存在于默认分支 (通常是 `main` 或 `master`)** 中，才能在 UI 上显示手动运行按钮。仅存在于非默认分支 (`dev`) 时不可见。

**解决方案**:
*   将 `dev` 分支的更改合并到 `main` 分支并推送。
*   推送后，GitHub Actions 界面即现出 "CI - Build Base Image" 选项，可点击 "Run workflow" 手动构建基础镜像。

---

## 3. 总结 CI/CD 修复流程

1.  **修正代码**: 确保 Dockerfile 支持动态用户名。
2.  **构建基础**: 先运行 `CI - Build Base Image`，成功推送 `liujiankang123/cloud-robot-viz-base:latest`。
3.  **构建应用**: 再次运行 `CD - Build & Deploy`，此时它就能成功拉取到刚构建好的基础镜像，完成后续的应用构建与部署。
